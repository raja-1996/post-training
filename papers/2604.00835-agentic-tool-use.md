## Definitions and scope

- **LLM agent:** an LLM augmented with planning, memory, and action capabilities, with the LLM acting as the central reasoning core.
- **Tool use** = the mechanism that operationalizes "action." Examples of tools: APIs, databases, code interpreters, search engines, robots (SayCan), translators, calculators, GUIs.
- The survey distinguishes tool use from earlier static augmentation (e.g., pure RAG) by requiring *interactive* invocation where the model decides when and how to call an external resource and incorporates returned observations.
- "Agentic tool use" is positioned as the union of *function calling* (low-level invocation correctness), *tool selection*, *planning/decomposition*, and *long-horizon interaction*.

## Taxonomy / organizing framework

The paper's central contribution is a **three-paradigm taxonomy** organized by the *source of the optimization signal* used to teach tool use:

| Paradigm | Signal | Weights touched? | Core challenge |
|---|---|---|---|
| I. Prompting as Plug-and-Play | in-context control (instructions, demos, observations) | No (frozen LLM) | Brittle on complex multi-step tasks; high latency/token cost |
| II. Supervised Tool Learning | labeled/synthetic SFT supervision | Yes (full SFT) | Constructing large, high-quality tool-use datasets |
| III. Reward-Driven Tool Policy Learning | environmental reward feedback (RL) | Yes (RL fine-tuning) | Credit assignment in long tool-call chains; on-policy data |

Evaluation is treated as an **orthogonal cross-cutting dimension** with three layers (Section 6):
1. **Tool usage correctness** (function-call validity, retrieval/selection, when-to-use decisions, compositional invocation).
2. **Task completion** (math/knowledge QA, programming, scientific/professional workflows).
3. **Tool-driven interaction** (web, computer/OS, safety/reliability).

The paper argues the three paradigms are *complementary* and that production-grade systems will combine all three.

## Comparison with prior surveys (Table 1)

Compared along six axes (chronological evolution, paradigm taxonomy, tool coverage, training data, evaluation landscape, open challenges):
- Zhao et al. [193], Wang et al. [142], Mialon et al. [89] — strong on chronology and evaluation but weak on taxonomy/training data.
- Qu et al. [113], Shen [123], Chen et al. [20] — strong on taxonomy/tool coverage but weak on chronological evolution.
- This survey claims to cover all six.

## Paradigm I: Prompting as Plug-and-Play (Section 3)

Frozen LLM coordinates tools via prompts, in-context examples, and observation feedback. Three sub-paradigms:

### 3.1 Interleaved reasoning and action
Multi-turn alternation between thought and action; observation drives next step.
- **ReAct [174]** — canonical Thought–Action–Observation loop.
- **Reflexion [125]** — verbal self-reflection from failures.
- **Self-Refine [86]** — iterative self-improvement of intermediate reasoning/outputs.
- **Chain-of-Verification [32]** — model-generated verification questions before final answer.
- **CRITIC [40]** — externally tool-grounded self-correction.
- **LATS [197]** — ReAct + tree search for deliberate trajectory exploration.
- Related prompting (not strictly tool-use but reinforces the paradigm): **Self-Consistency [148]**, **Least-to-Most [199]**, **Take a Step Back [194]**.

### 3.2 Decoupled planning and execution
Planner first produces a structured plan; executor then carries out tool calls without per-step replanning.
- **ReWOO [164]** — planner/worker with variable assignments, reduces interaction overhead.
- **LLMCompiler [64]** — dependency-aware computation graph with parallel tool execution.
- **AdaPlanner [128]** — adaptive plan revision under feedback.
- **Plan-and-Solve Prompting [143]**.
- **Skeleton-of-Thought [94]** — coarse outline then expand.
- **PEARL [129]** — plan-and-execute for long documents.
- **ART [100]** — retrieve programs from external task library.
- **Chameleon [82]** — modular execution pipelines.
- **HuggingGPT [122]** — LLM as planner over diverse expert models.
- **TaskMatrix.AI [76]** — connect foundation models with millions of APIs.

### 3.3 Program-aided reasoning
External executable code is *part of* reasoning, not just a utility.
- **PAL [37], PoT [23]** — Python externalization of computation.
- **Logic-LM [99]** — translation to symbolic formulations solved by logic solvers.
- **Chain of Code [71]** — mix runnable code with LM-emulated pseudo-code.
- **ViperGPT [131]** — code to call vision APIs.
- **Code as Policies [75]** — compile NL to robot policy code via predefined APIs.
- **MathPrompter [53]** — generate multiple candidate Python functions; check agreement.
- **LATM [14]** — agents create reusable Python tools for unseen tasks ("Large Language Models as Tool Makers").
- **CREATOR [108]** — formalize tool creation as a separate stage from decision execution.

## Paradigm II: Supervised Tool Learning (Section 4)

Internalize tool use into model parameters via SFT on labeled/synthetic supervision.

### 4.1 Self-supervised data generation
- **Toolformer [121]** — self-annotation with the loss-reduction filter (Equation 1):
  $$\Delta\ell_i(c, r) = \ell_i^- - \ell_i^+ = -\log p_\theta(x_i \mid c) + \log p_\theta(x_i \mid c, r)$$
  Keep a tool call only if its returned result $r$ reduces next-token loss given context $c$. Central intuition: tool calls earn their place by improving prediction, not by mere syntactic validity.
- **ToolAlpaca [133]** — teacher-generated synthetic tool-use demonstrations.
- **AutoAct [110]** — auto-synthesized planning trajectories for tool-augmented agents.
- **APIGen [81]** — multi-stage pipeline with executable verification for diverse function-calling data.
- **ToolACE [79]** — self-evolving API augmentation + dialogue synthesis + hierarchical validation.
- **Confucius [38]** — iterative tool learning via introspective feedback and easy-to-difficult curriculum.

### 4.2 Large-scale instruction tuning
Move from small predefined tool sets to thousands of real-world APIs.
- **ToolLLM [112]** + **ToolBench** — instruction tuning over 16,000+ RESTful APIs.
- **Gorilla [104]** — retrieval-aware training conditioning calls on retrieved documentation.
- **API-Bank [73]** — planning + retrieval + response generation.
- **AnyTool [34]** — hierarchical retrieval over large toolsets.
- **EasyTool [181]** — token-efficient transformed tool documentation.
- **Octopus v2 [22]** — lightweight on-device function-argument mapping for latency.
- **ToolkenGPT [43]** — tools as parametric token embeddings (no vocab blow-up).
- **xLAM [187]** — high-performance function calling over large action spaces.
- **Granite-Function-Calling [1]** — multi-task learning of parameter prediction.
- **Functionary [88]** — tight integration of chat and function invocation.
- **NexusRaven [127]** — robust zero-shot generalization from descriptions.

### 4.3 Process-oriented and alignment-focused tuning
Optimize the reasoning *process* and align with safety/preferences, not just final call correctness.
- **FireAct [16]** — SFT on reasoning trajectories (vs. input-output pairs).
- **ToRA [41]** — interleaved reasoning-action trajectories with explicit rationales.
- **Synapse [195]** — distill trajectories into exemplars.
- **Masked Thought [17]** — train to recover masked parts of reasoning chains; discourages surface pattern matching.
- **Agent-FLAN [26]** — decompose agent capability into format-following + reasoning; use negatives to reduce hallucinated actions.
- **RAFT [189]** — distinguish relevant from irrelevant retrieved docs.
- **ToolAlign [27]** — H2A principle: helpfulness, harmlessness, autonomy.
- **ToolSword [177]** — adversarial scenarios (e.g., prompt injection) for safety alignment.
- **RoT [178]** — robustness under noisy / imperfect tool documentation.
- **MetaTool [51]** — train models to assess whether a tool is needed at all.
- **Lemur [166], AgentTuning [183]** — preserve generalist competence alongside agentic SFT.

## Paradigm III: Reward-Driven Tool Policy Learning (Section 5)

Optimize tool interaction through environmental feedback / RL; treats tool use as sequential decision making.

### 5.1 Strategic decision optimization
Optimize the high-level decision: *when* to invoke a tool vs. rely on internal knowledge.
- **PPO clipped surrogate objective** (Equation 2/3) is the standard policy-gradient backbone:
  $$\mathcal{L}^{\text{CLIP}}(\theta) = \hat{\mathbb{E}}_t\left[\min\bigl(r_t(\theta)\hat{A}_t,\, \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon)\hat{A}_t\bigr)\right],$$
  with $r_t(\theta) = \pi_\theta(a_t\mid s_t)/\pi_{\theta_\text{old}}(a_t\mid s_t)$. Here $\hat{A}_t$ is the advantage of a tool-use action.
- **ReTool [35]** — RL to decide when to use external tools; reward favors correctness while discouraging unnecessary calls.
- **ToolRL [107]** — carefully scaled outcome-based rewards suffice; "Reward is All Tool Learning Needs."
- **ToolExpander [19]** — dynamic multi-round hard-sampling for weak LLMs; replaces persistent failure cases with stronger demos as a curriculum.
- **Agent0 [160]** — agent as both curriculum designer and executor via Ambiguity Dynamic Policy Optimization (ADPO); improves with no external data.
- **Arcee [144]** — Large Action Model trained with RL for enterprise workflows.

### 5.2 End-to-end policy learning for multi-turn reasoning
Optimize full reasoning-action policy across long trajectories; handle credit assignment.
- **Search-R1 [59]** — RL induces query reformulation and information synthesis from sparse outcome rewards.
- **SimpleTIR [167]** — unified end-to-end RL loop over reasoning + tool execution.
- **Retroformer [175], TRICE [109]** — retrospective learning + credit assignment over long tool chains.
- **AgentPRM [158]** — adapt process reward modeling to agentic tasks; evaluates intermediate step promise and progress.
- **LeTS [188]** — combine process-level + outcome-level rewards for joint reasoning/retrieval.
- **RAGEN [151]** — self-evolving agents from reflective revision data over prior trajectories.
- **MATPO [91]** — Multi-Agent Tool-Integrated Policy Optimization with role-aware planner-worker coordination; distributes rewards across distinct functions.
- **Agent-R [180]** — treat reflection as an explicit action inside the RL loop.

### 5.3 Holistic and multimodal agentic frameworks
Optimize the whole agent system, including memory and tool selection; extend to GUI / multimodal.
- **VerlTool [56]** — holistic framework; agent system (memory module + tool selector) as a unified policy updated jointly.
- **DeepAgent [74]** — autonomous memory folding compresses past interactions into structured episodic memory.
- **AgentRL [186]** — scalable multi-turn multi-task training via async generation-training pipeline, unified function-call API, centralized environment controller.
- **RLFactory [15]** — plug-and-play infra for simulating tool environments and trajectory-level rewards.
- **Agent Q [106]** — guided MCTS + self-critique; learns from both successful and failed trajectories via self-play in web environments.
- **SEAgent [130]** — computer-use agent with self-evolution that autonomously explores OS and curates its own instruction-tuning data.
- **DigiRL [6]** — offline-to-online RL for in-the-wild Android device-control agents.
- **GEPO [179]** — dynamically constructs state-transition graph from experience; uses graph centrality to guide exploration.
- **MGPO [50]** — high-resolution visual reasoning via multi-turn grounding-based RL.
- **OS-Copilot [157]** — generalist computer-use framework.
- **Cradle [132]** — general computer control via screen observation + keyboard/mouse emulation.
- **ToolRM [2]** — outcome reward models specialized for tool-calling LLMs.

- **ToolAlpaca [133]** — synthetic teacher-generated tool-use demonstrations for open small models.
- **APIGen [81]** — automated pipeline producing *verifiable* (executable) and diverse function-calling data.
- **ToolACE [79]** — self-evolving API augmentation + dialogue synthesis + hierarchical validation.
- **AutoAct [110]** — automatic planning-trajectory synthesis for tool-augmented QA agents.
- **Confucius [38]** — introspective-feedback synthesis with curriculum.
- **GPT4Tools [170]** — specialized data generation.
- **Toolformer [121]** — self-supervised annotation via Eq. (1) loss-reduction filter.
- **Agent-FLAN [26]** — synthetic data decomposed into format-following + reasoning, with negative samples.
- **AgentTuning [183]** — agent-oriented supervision mixed with general conversational data.
- **RAFT [189]** — synthetic retrieval-augmented training mixing relevant/irrelevant docs.
- **SEAgent [130]** — agent autonomously explores OS and curates its own instruction-tuning data.
- **RAGEN [151]** — generates reflective revision data from prior trajectories.

The paper's narrative is that data pipelines moved from *trajectory generation* → *function-call quality and verifiability* (APIGen, ToolACE) → *curriculum / introspection* (Confucius) → *self-evolution* (SEAgent, Agent0).

## Tool-call formats and schemas

The paper does not standardize on one schema but discusses the trends:
- **Schema-constrained function calling** — formal JSON / function signature outputs evaluated by BFCL [103].
- **REST API calls** — ToolLLM/ToolBench's primary format.
- **Code as the action space** — PAL, PoT, ViperGPT, Code as Policies, CodeAct [145].
- **Token-level tool embeddings** — ToolkenGPT [43] represents tools as embedded tokens instead of textual schemas.
- **Compressed / token-efficient docs** — EasyTool [181].
- **Standardized inter-agent protocol** — **Model Context Protocol (MCP) [49]** as the emerging universal interface, reducing integration from quadratic to linear complexity.

## Reward design for tool use

Reward designs the paper highlights:
- **Outcome-only reward** — final task success / correctness; can suffice when carefully scaled (ToolRL [107]) and powers Search-R1 [59].
- **Selectivity reward** — reward correct outcomes while penalizing unnecessary tool calls (ReTool [35]).
- **Process reward modeling (PRM)** — evaluate intermediate steps for promise and progress (AgentPRM [158]).
- **Process + outcome blend** — LeTS [188] combines both for joint reasoning/retrieval.
- **Trajectory-level rewards** — infra in RLFactory [15] and VerlTool [56].
- **Specialized tool-outcome reward models** — ToolRM [2] for tool-calling efficiency and correctness.
- **Curriculum / hard-sampling reward shaping** — ToolExpander [19] replaces persistent failures with stronger demos; Agent0 [160] auto-curriculum via ADPO.
- **Role-distributed rewards** — MATPO [91] for planner-worker coordination.
- **Reflection-as-action** — Agent-R [180] embeds critique inside the RL loop.

## Multi-step / multi-tool / planning

- Decoupled planning (ReWOO, LLMCompiler, AdaPlanner) reduces per-step deliberation overhead.
- Compositional / nested calls evaluated by **NESTFUL [7]** (output of one call feeds the next) and **NexusRaven [127]**.
- Long tool-chain credit assignment addressed by **TRICE [109]**, **Retroformer [175]**, **AgentPRM [158]**.
- Multi-agent coordination handled by **MATPO [91]** (role-aware), **MetaGPT [48]**, **AutoGen [155]**, **OpenAI Swarm [116]**.
- Tree-style planning: **LATS [197]**, **Tree of Thoughts [173]**, **Agent Q [106]** (MCTS + self-critique).

## Memory and long-horizon agency

- **MemGPT [98]** — OS-style memory management for long context.
- **Generative Agents [102]** — memory for sustained behavior in agent simulations.
- **DeepAgent [74]** — autonomous memory folding to compress past interactions into episodic memory.
- **Titans [8]** — neural long-term memory module learning to memorize at test time, "needle-in-a-haystack" retrieval over massive histories without retraining.
- **System 2 Attention [154]** — dynamically regenerates context to focus on relevant info.
- **Trove [150]** — continuous evolution of an agent's tool library.
- **Voyager [141]** — incremental skill library acquisition in Minecraft.

## Future directions / open problems (Section 7)

Five trajectories highlighted:

1. **Standardization and the MCP revolution.** Move toward **Model Context Protocol (MCP) [49]** as universal interface, decoupling agent from tool implementation. Build tool once, use it everywhere. Quadratic → linear integration complexity. Standard orchestration patterns: **OpenAI Swarm [116]**, **AgentKit [156]** (unified framework for "reinforcement fine-tuning" of agentic workflows). Tool libraries become *dynamic protocol-compliant marketplaces*.

2. **Multimodal and embodied foundation models.** Agents perceive and manipulate GUIs directly rather than relying on structured function calls. "Agentic Foundation Models" pretrained for action (Magma [169], ShowUI [78], OmniParser [84], Agent-S [3]). Move from patching LLMs with vision encoders to native VLA models.

3. **Agentic self-consistency and continuous evolution.** Test-time scaling and lifetime learning. Search-R1 [59] for internalized reasoning + retrieval; **Titans [8]** for test-time-trained long-term memory; **System 2 Attention [154]** for dynamic context regeneration; **Trove [150]** for evolving tool libraries. **Character-R1 [134]** explores role-aware reasoning and persona consistency.

4. **Higher safety and trustworthiness.** Defenses against indirect prompt injection (InjEcT-Agent [184]); programmable safety layers (**NeMo Guardrails [118]**); adversarial-input detection (**PromptArmor [124]**); avoiding deceptive alignment / reward hacking (**Sleeper Agents [52]**); verifiable audit trails and sandboxed execution.

5. **Human–agent symbiosis and workflow reshaping.** Multi-agent collaboration (**MetaGPT [48]**, **AutoGen [155]**, **OpenDevin [147]**), system-level orchestration of specialized agents [198], **skill-centric** abstractions over fully autonomous monolithic agents. Counter-trend: **Agentless [159]** shows simple structured pipelines can outperform complex long-horizon agents when task structure is clear.

Earlier in the paper, additional open problems are surfaced:
- **Long-horizon credit assignment** for multi-tool chains.
- **Scalable tool generalization** across evolving, heterogeneous tool ecosystems.
- **Autonomous tool creation and self-extension** (LATM, CREATOR, Trove).
- **Alignment-aware deployment** integrating safety/compliance.
- **Cost / latency trade-offs** in prompting-based agents.
- **Balancing tool specialization vs. generalist competence** (recurring concern of Lemur, AgentTuning).

## Notable formulations / equations

The survey gives two explicit equations:

**Equation 1 — Toolformer's tool-call utility filter:**
$$\Delta\ell_i(c, r) = -\log p_\theta(x_i \mid c) + \log p_\theta(x_i \mid c, r)$$
A candidate tool call with response $r$ is kept iff inserting it improves prediction of subsequent tokens $x_i$ given context $c$. Captures the "self-supervised" criterion for tool-use annotation.

**Equations 2–3 — PPO clipped surrogate (the default RL objective referenced in Paradigm III):**
$$\mathcal{L}^{\text{CLIP}}(\theta) = \hat{\mathbb{E}}_t\bigl[\min\bigl(r_t(\theta)\hat{A}_t, \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon)\hat{A}_t\bigr)\bigr], \quad r_t(\theta) = \frac{\pi_\theta(a_t \mid s_t)}{\pi_{\theta_\text{old}}(a_t \mid s_t)}.$$
With $\hat{A}_t$ as the advantage of a tool-use action. Clipping stabilizes updates against large policy deviations.

The paper does **not** include the GRPO objective explicitly, nor an explicit process-reward-modeling formula, though both are referenced conceptually (AgentPRM [158], LeTS [188]).

## Notes on coverage

- The paper is itself a survey, so this summary is a compression of a compression. Method-level mechanics for each cited system are not deep — for any specific method (e.g., AgentPRM's exact reward shape, MATPO's role assignment, ToolACE's hierarchical validation), the original paper should be consulted.
- The survey emphasizes prompting / SFT / RL paradigms, evaluation layers, and a few representative future directions. It does **not** deeply cover: GRPO-style objectives, specific frontier-model post-training pipelines (Claude / GPT / Kimi K2), or detailed dataset statistics.
- Figures 1–5 are conceptual diagrams; their exact graphical layout is not reproduced here but their semantic content is captured in this markdown.
- The equation set is limited to Toolformer's loss-reduction filter and PPO's clipped surrogate — these are the only two formulas explicitly given in the paper.
- Numbers in square brackets throughout are the paper's own reference indices and can be cross-checked in the PDF's bibliography starting at page 21.
